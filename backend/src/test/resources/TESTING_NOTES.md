# Заметки по автоматическим и ручным тестам валидации ВКР

## Зачем два уровня проверок

- **Юнит-тесты** проверяют чистую логику чекеров на синтетических `ParagraphInfo` / настройках страниц без загрузки Word. Они быстрые, детерминированные и ловят регрессии в форматировании сообщений и ветвлениях (например, граничные значения полей и отступов).
- **Интеграционные тесты** (`validation/*`) гоняют полный пайплайн на реальном `.docx` из `test-input/`, когда файл существует.
- **Ручная проверка** на ваших документах нужна для **ложных срабатываний** (особенности вёрстки Word, поля, колонтитулы, списки) и для **скорости** на больших файлах — юнит-тесты этого не заменяют.

После доработок чекеров имеет смысл прогонять `gradlew test` и затем закидывать типичные и «пограничные» `.docx` через ваш отладочный запуск (например, `runDocxDebug` / `DocxLoadDebugMain`), сравнивая `output/right.txt` и `output/wrong.txt`.

## Матрица: файл теста → что проверяется → кейсы → почему именно так

| Тестовый класс | Смысл | Основные кейсы | Почему такие кейсы |
|----------------|--------|----------------|-------------------|
| `BodyParagraphRulesTest` | Кто считается «основным текстом» для ФТ-8/ФТ-9 | Обычный абзац; заголовок с outline; таблица; подпись к рисунку; строка похожая на оглавление | Границы правила — место, где чаще всего ложные срабатывания в реальных документах |
| `Ft4RequiredSectionsCheckerTest` | Обязательные разделы и главы | Пустой документ; минимальный набор заголовков уровня 0 | Пустой список даёт все типы отсутствующих разделов; полный комплект — регрессия «всё нашли» |
| `Ft5SectionNumberingCheckerTest` | Нумерация между введением и заключением | Нет введения/заключения; лишняя точка в номере `1.1.` | Повторяет ранний выход и типичную ошибку оформления по методичке |
| `Ft6SectionStartCheckerTest` | Новая страница перед обязательными блоками | Начало на той же странице, что конец предыдущего абзаца; корректный разрыв страницы | Проверяет только эвристику по `pageIndex` / `pageEndIndex` (без OOXML в тесте) |
| `Ft8MainFontCheckerTest` | Шрифт, кегль, цвет основного текста | Норма; нарушение по run-флагам; не TNR; мелкий кегль; нечёрный цвет; хелперы `isTimesNewRoman` / `isBlackColor` | Покрывает разные ветки: сводка по абзацу vs детекция по прогонам |
| `Ft9MainParagraphCheckerTest` | Интервал 1,5; красная строка; по ширине | Норма; неверный интервал; отступ; не BOTH | Типичные отклонения от п. 4.2; без загрузки документа |
| `Ft10PageMarginsCheckerTest` | Поля страницы | Левое поле не 30 мм; норма по секции | Секция с явными `PageMargins` вместо полного парсинга |
| `Ft11HeadingFormattingCheckerTest` | ФТ-11 (заголовки) | Нет ни одного кандидата с outline/Heading | Дымовой тест ветки «проверка пропущена»; полный ФТ-11 в юнитах тяжёлый |
| `Ft12PageNumberingCheckerTest` | Нумерация страниц (подвал, PAGE) | Нет `pageSettings`; нет `numbering`; нет PAGE в подвале; минимально корректный `PageNumberingInfo` | Быстрые сценарии без полного документа с колонтитулами |
| `ChecksConfigurationLoaderTest` | Загрузка `checks-config.json` | Парсинг, дубликаты id, параметры `ft21` | Контракт конфигурации с продом |
| `Ft13–Ft21*Test` (существующие) | Подписи, таблицы, формулы, списки, библиография и т.д. | См. сами классы | Сохраняют поведение по уже существующим ФТ |
| `ValidationPipelineIntegrationTest` | Полный прогон на `test.docx` | Включён, если файл есть (`@EnabledIf`) | Регрессия сквозного пути |
| `ValidationPerformanceSmokeTest` | Время полного прогона | Верхняя граница времени на том же файле | Раннее обнаружение сильного замедления |
| `CorruptedDocumentHandlingTest` | Повреждённый вход | не-OOXML байты → `ValidationException` | Пользователь не должен видеть сырой POI-стек |
| `OfficeMathDetectorTest` | Детектор формул в XML | Фрагменты OOXML | Изоляция утилиты от парсера документа |
| `BackendApplicationTests` | Поднятие Spring-контекста | Smoke | CI |

## Вспомогательный код

- `com.example.backend.testsupport.ParagraphFixtures` — собирает «типичный» абзац основного текста с полями, подходящими под `BodyParagraphRules`, чтобы не дублировать длинные `builder()` в каждом тесте.

---

## Что смотреть при ручной проверке на `.docx`

1. **Ложные срабатывания** — замечания, которые не соответствуют методичке при визуально корректном документе (часто: страницы без `lastRenderedPageBreak`, сложные колонтитулы, поля в шаблоне).
2. **Скорость** — время полного прогона на больших документах; при необходимости сравнивайте с `ValidationPerformanceSmokeTest` как ориентиром на тестовом файле.

Этот файл можно обновлять при добавлении новых тестовых классов или изменении сценариев ручной проверки.
