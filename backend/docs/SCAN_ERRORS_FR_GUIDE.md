# Замечания сканера: как исправить в Word и связь с требованиями

Документ основан на типичных сообщениях валидации ВКР и на том, как они формируются в коде (`Ft4`, `Ft9`, `Ft10`, `Ft12`, `Ft14`, `Ft18` и др.).

---

## 1. Нумерация страниц (ФТ-12, п. 4.3.1)

### 1.1. «Отдельный подвал первой страницы без поля PAGE»

**Что требует методичка (п. 4.3.1):** сквозная нумерация, номер внизу по центру, без точки после номера.

**Что видит проверка:** в документе включён **особый подвал для первой страницы** (или иной вариант «первая страница — другой подвал»), и в XML этой части подвала **нет поля номера страницы** (`PAGE`). На титуле часто номер сознательно скрывают — тогда сообщение формулируется как «сквозной номер снизу может отсутствовать», но отдельный подвал без `PAGE` всё равно фиксируется как отклонение от единого шаблона.

**Как исправить в Word:**

- Вставка → Номер страницы → внизу страницы → простой номер по центру **в том подвале, который действует на первую страницу**, либо  
- Отключить «особый подвал для первой страницы» и использовать один подвал для всех страниц (если по вашему вузу титул допускает номер), либо  
- Оставить отдельный подвал, но **вставить в него то же поле PAGE** (пусть визуально скрыто/в колонтитуле с полем «скрыть на первой» — зависит от шаблона Word).

**Функциональное требование:** п. 4.3.1 — единообразие нумерации и размещения внизу по центру.

---

### 1.2. «Отдельный подвал чётных страниц без поля PAGE»

**Смысл:** включены **разные колонтитулы для чётных/нечётных страниц**; в подвале чётных страниц нет поля `PAGE`.

**Как исправить:** в режиме редактирования подвала для **чётных** страниц вставить номер страницы (поле PAGE) **по центру**, как в подвале нечётных, либо убрать разделение чёт/нечёт, если методичка требует один подвал.

**ФТ:** снова п. 4.3.1.

---

### 1.3. «Абзац с полем PAGE в нижнем колонтитуле не выровнен по центру»

**Что это значит простыми словами:** номер страницы в подвале **есть**, но в разметке OOXML абзац (или блок с номером) **не помечен как выровненный по центру**. Проверка смотрит:

- в сыром XML подвала — наличие выравнивания `w:jc` со значением `center`;
- для классического поля в абзаце — `ParagraphAlignment.CENTER` у абзаца в подвале.

**Типичные причины ложного или спорного срабатывания:**

- Номер вставлен через **таблицу** в подвале (ячейка с выравниванием «влево»);
- Номер в **конструкции SDT / галереи** Word, где центрирование задано иначе, чем ожидает эвристика;
- Несколько подвалов: в одном центр есть, в другом — нет, итог сводится логикой «все с PAGE должны быть по центру».

**Как исправить:** открыть подвал → выделить абзац с номером → «Главная» → **по центру**; либо упростить подвал (один абзац, только поле PAGE по центру).

**ФТ:** п. 4.3.1 — внизу **по центру**.

## 2. Подпись к таблице (ФТ-14, п. 4.6.3)

**Сообщение:** не шаблон «Таблица N – Название», найдено «Таблица 1: …».

**Требование:** после слова «Таблица» — **номер арабскими цифрами**, затем **тире** (дефис/длинное тире по методичке) и название; двоеточие вместо тире не подходит под ожидаемый шаблон.

**Как исправить:** заменить `Таблица 1:` на `Таблица 1 – …` (или тот вид тире, который прописан у вас в Приложении 1 / п. 4.6.3).

**Связь с «Согласуйте оглавление… (п. 3.4)»:** это **общая подсказка** в отчёте рядом с блоком «Содержание / заголовки»; к самой таблице она относится косвенно (если заголовок таблицы не попадает в оглавление как заголовок — проверьте стили).

## 3. Список терминов (ФТ-18, п. 4.9.2)

### 3.1. Выравнивание не по левому краю (фактически BOTH)

**Смысл:** для варианта «список абзацами» строки глоссария должны быть **по левому краю**. У вас в OOXML у абзаца стоит выравнивание **по ширине** (`BOTH` / justify).

**Как исправить:** выделить абзацы списка терминов → выравнивание **по левому краю**.

---

### 3.2. «В конце строки не должно быть знаков препинания»

**Смысл:** по п. 4.9.2 определение не должно заканчиваться точкой, запятой и т.п. (в конце **всей** строки термин–определение).

**Как исправить:** убрать финальную точку/запятую в тексте определения.

---

## 4. «В тексте не найден обязательный раздел «СОДЕРЖАНИЕ»» (ФТ-4, п. 3.2, 4.4.1)

### Как **сейчас** ищется «СОДЕРЖАНИЕ» в коде (важно)

Ищется **не «сверху документа»** и **не первый абзац**. Алгоритм:

1. Просматриваются **все абзацы по порядку** от начала к концу.
2. Учитываются только те, у которых **`outlineLevel == 0`** (в терминах OOXML: уровень структуры **0** — это верхний уровень заголовка в XML, соответствует «Заголовок 1» / первому уровню схемы, если стиль так настроен).
3. Текст заголовка после нормализации (пробелы, верхний регистр) должен **точно совпасть** с `СОДЕРЖАНИЕ`.

**Почему «в начале есть, а сканер ругается»:**

- Заголовок набран обычным текстом или стилем **без** привязки к уровню структуры 0 → в модели абзаца `outlineLevel` = `null` → заголовок **игнорируется** (так же отсекаются строки поля оглавления TOC без нужного outline).
- Стиль даёт **другой** уровень (например, не 0 в том виде, как его читает парсер).
- Текст не совпадает: лишние символы, другой язык, «Содержание:» в одной строке с чем-то ещё так, что нормализация не даёт ровно `СОДЕРЖАНИЕ`.

**Как исправить:** пометить строку «СОДЕРЖАНИЕ» как **заголовок 1-го уровня структуры** (в Word: стиль «Заголовок 1» или задайте для абзаца уровень структуры 1 в диалоге абзаца — в OOXML это обычно отображается как нужный `outlineLvl` для проверки **0** в коде; если после правки замечание останется, имеет смысл смотреть фактический `w:outlineLvl` в XML).

**ФТ:** п. 3.2 (обязательные разделы), п. 4.4.1 (прописные для таких заголовков).

## 5. «Неверный межстрочный интервал» при сообщении про выравнивание (ФТ-9, п. 4.2)

В интерфейсе отчёта категория может называться шире, но текст замечания относится к **ФТ-9**:

- **Ожидается по ширине, фактически LEFT** — основной текст: выравнивание **по ширине** (Justify / BOTH в OOXML).
- **Ожидается 1,5, фактически 0** — у абзаца в XML не задан межстрочный интервал (часто пустой абзац, заголовок, подпись) или интерпретируется как 0; для **основного текста** задайте **1,5** в параметрах абзаца.
- **По центру вместо по ширине** — тот же п. 4.2: для рассматриваемого абзаца как «основного текста» нужна **ширина**, не центр.

**Как исправить:** для абзацев **основного текста** (не заголовки, не подписи к рисункам/таблицам по правилам чекера): межстрочный **1,5**, выравнивание **по ширине**.

**Примечание:** абзацы в таблицах, формулы, заголовки по outline и подписи «Рисунок N…» / «Таблица N…» для ФТ-9 могут **не** проверяться как основной текст — если замечание всё же есть, значит абзац попал в фильтр «основной текст».

**Термины и определения (п. 4.9.2):** абзацы между заголовком раздела из настройки ФТ-18 (`sectionTitle`, по умолчанию «ТЕРМИНЫ И ОПРЕДЕЛЕНИЯ») и следующим заголовком уровня 0 **не проходят ФТ-9** (для них по методичке допустимо выравнивание по левому краю и иные правила из ФТ-18).

## 6. Неверные поля страницы (ФТ-10, п. 4.2)

Примеры из скана:

- **Левое:** ожидается **30 мм**, у вас **20 мм**.
- **Правое:** ожидается **10–15 мм**, у вас **20 мм**.
- **Нижнее:** ожидается **20 мм**, у вас **~30 мм**.

**Как исправить:** Макет страницы → Поля → задать **слева 3 см**, **справа 1–1,5 см**, **сверху и снизу по 2 см** (или точные мм по вашей методичке).

**Про «оценка стр. 1» и «абз. #0–#24»:** поля в Word задаются **на секцию**. Номер страницы в сообщении — **оценка по разрывам OOXML**, не «напечатанная» вёрстка Word. На точность влияет, сохранён ли документ с пересчётом вёрстки.

## 7. «Должен начинаться с новой страницы», но визуально всё на новой (ФТ-6, п. 4.4.2)

Сообщение вида: *«начало на стр. N, предыдущий абзац заканчивается на стр. N»* идёт из **`Ft6SectionStartChecker`**. Рядом в интерфейсе может быть подсказка про **п. 4.4.4** (оформление заголовков) — это **другая** проверка (ФТ-11); критерий «новая страница» задаётся только **ФТ-6**.

### Когда для сканера заголовок **действительно** «с новой страницы»

В коде сравниваются два числа из **OOXML**, а не экранная вёрстка Word:

- **`pageIndex`** у абзаца-заголовка — с какой **оценочной** страницы начинается абзац;
- **`pageEndIndex`** у **предыдущего непустого** абзаца — на какой странице этот абзац **заканчивается** (с учётом разрывов внутри абзаца).

Условие «раздел/глава начинается с новой страницы» считается выполненным только если:

```text
pageIndex(заголовок) > pageEndIndex(предыдущий абзац)
```

**Пример, когда замечания нет:** предыдущий текст заканчивается на стр. 3 (`pageEndIndex = 3`), заголовок начинается на стр. 4 (`pageIndex = 4`) → `4 > 3`.

**Пример, когда замечание есть:** и там, и там **4** → `4 > 4` ложно: в файле заголовок привязан к **той же** оценочной странице, на которой закончился предыдущий блок.

### Когда визуально «новая страница», а в файле — «та же страница»

Номера страниц для абзацев строятся по **`PageLocator`**: учитываются явные разрывы страницы (`w:br` типа page), маркеры **`w:lastRenderedPageBreak`** (как Word сохранил разбивку при последнем сохранении), разрыв секции «**следующая страница**». Это **не** то же самое, что «я вижу в режиме разметки/печати», если:

1. Документ открывали не в Word или сохраняли **без** пересчёта вёрстки — в XML **нет** `lastRenderedPageBreak` между нужными абзацами, и цепочка страниц «не двигается».
2. Перед заголовком нет **явного** разрыва страницы в потоке: отступами/«не помещается» Word переносит строку на новую страницу при отображении, но **между двумя абзацами** в XML нет ни разрыва страницы, ни маркера переноса в нужном месте — тогда оба абзаца могут получить одну и ту же оценку «стр. N».
3. «Предыдущий абзац» для проверки — это **последний непустой** абзац **перед** заголовком (пустые пропускаются). Если перед «ВВЕДЕНИЕ» идёт длинный абзац оглавления/текста, который в модели **заканчивается на той же странице N**, что и начало заголовка, получится как в сообщении: оба **N**.

### Что сделать на практике

1. Поставить курсор **перед** заголовком раздела → **Ctrl+Enter** (разрыв страницы) или «Макет» → «Разрывы» → **страница**. Так в XML появляется явный сигнал переноса.
2. Альтернатива: вставить разрыв в **конец предыдущего** абзаца (тогда разрыв оказывается в последнем прогоне предыдущего абзаца — так тоже корректно учитывается).
3. Открыть документ в **Word**, дать ему пересчитать вёрстку, **сохранить** (чтобы при необходимости появились `lastRenderedPageBreak`).
4. Для отладки в проекте смотрите лог **`DocxLoadDebugMain`**: там выводится последовательность `pageIndex` / `pageEndIndex` по абзацам — по ней видно, где обрывается «страница» в модели.

**Загрузчик:** для номера страницы «начала» абзаца учитываются разрывы в **первых прогонах до первого непустого текста** (`PageLocator.paragraphStartPageForContent`), чтобы разрыв **в начале** абзаца с заголовком (частый случай после Ctrl+Enter перед заголовком) не давал ложного «начало и конец на одной стр. N» в ФТ-6.

### Пример из реального лога (`right.txt`, образец ВКР)

В диагностике абзацев:

- **`Paragraph[42]`** (последняя строка терминов): `pages=3-4` — абзац **начинается** на стр. 3, **заканчивается** на стр. 4 (перенос внутри абзаца).
- **`Paragraph[43]`** «ВВЕДЕНИЕ»: `pages=4-4` — заголовок **начинается** на стр. 4.

Для ФТ-6 сравниваются `pageEndIndex` предыдущего непустого абзаца (4) и `pageIndex` заголовка (4). Условие «новая страница»: **стр. начала заголовка > стр. конца предыдущего** — здесь **4 > 4** ложно, поэтому замечание **формально верное по XML**, даже если визуально перед «ВВЕДЕНИЕ» кажется «новый лист»: последняя строка терминов уже заняла стр. 4. Аналогично для глав **1. …**, **2. …** и т.д., если предыдущий блок текста **заканчивается на той же** оценочной странице, что и заголовок главы.

**Что сделать:** вставить **явный разрыв страницы** перед нужным заголовком (или перенести разрыв в конец предыдущего абзаца), чтобы в модели номер страницы **начала** заголовка стал **больше**, чем `pageEndIndex` предыдущего абзаца; либо укоротить предыдущий абзац, чтобы он не уходил на следующую страницу.

**ФТ:** п. 4.4.2 (начало с новой страницы для перечисленных блоков).

## 8. Оглавление: «нет заголовка … СПИСОК ИСПОЛЬЗУЕМЫХ ИСТОЧНИКОВ» (ФТ-7 / согласование TOC)

Если в тексте заголовок назван **«СПИСОК ИСПОЛЬЗОВАННЫХ ИСТОЧНИКОВ»** (как в ФТ-4 по умолчанию), а в поле оглавления автоматически попала строка **«СПИСОК ИСПОЛЬЗУЕМЫХ …»** (другая буква в слове), проверка сочтёт, что **в теле нет** заголовка, совпадающего с тем, что в TOC.

**Как исправить:** привести к **одному** написанию заголовок в тексте и обновить оглавление (ПКМ по полю TOC → обновить), либо исправить опечатку в исходном заголовке.
