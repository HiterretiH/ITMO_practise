# Материал для отчёта по практике: бэкенд загрузки .docx и ядро проверок ВКР

Документ предназначен как **подробный сырой материал** для последующей литературной обработки (в т.ч. нейросетью): что реализовано, **почему** так устроено, какие ограничения у подхода и какие **инженерные проблемы** решались в процессе.

Исходная формулировка этапа:

> Реализовать бэкенд-модуль загрузки и парсинга .docx файлов. Разработать ядро проверки структуры документа (наличие обязательных разделов, нумерация, начало с новой страницы) и базовых параметров форматирования текста (шрифты, отступы, поля). Представить результат работы преподавателю по окончании этапа.

---

## 1. Общая архитектура решения

### 1.1. Роль компонентов

- **`DocxLoadService`** — единая точка загрузки: принимает поток, валидирует расширение/тип, открывает документ через **Apache POI** (`XWPFDocument` для OOXML), обходит тело документа и собирает **нормализованную модель** `DocumentStructure`.
- **`ChecksConfigurationLoader`** — читает **`checks-config.json`** (список правил ФТ-4 … ФТ-21, флаги `enabled`, параметры норм).
- **`CheckSession`** — на время прогона проверок кладёт в потоковый контекст параметры текущего правила (чтобы чекеры не дублировали чтение JSON).
- **`VkrChecksRunner`** — последовательно вызывает нужный класс-проверку по `id` правила; возвращает список **`CheckExecutionResult`** (id, заголовок, список текстовых замечаний).
- **`CheckEngineService`** — склеивает загрузку и прогон; при необходимости мапит строки замечаний в структурированные **`ErrorItem`** через **`ValidationIssueMapper`**, считает статус (`passed` / `warning` / `failed`).

Таким образом, **парсинг отделён от бизнес-правил**: новая проверка = новый класс + строка в `checks-config.json` + ветка в `switch` раннера (или рефакторинг на регистрацию по id — на текущем этапе явный `switch`).

### 1.2. Почему не «вёрстка Word внутри Java»

Формат **.docx** — это ZIP с XML (OOXML). Полноценная вёрстка страниц, как у редакторов, в бэкенде **не выполняется**. Вместо этого:

- извлекаются **явные данные OOXML** (абзацы, свойства, разрывы, секции, поля);
- где нужно «приблизить» страницу, используется **эвристика** по разрывам (`w:br` типа page, `lastRenderedPageBreak`, разрыв секции «следующая страница») — класс **`PageLocator`**.

Это **осознанный компромисс**: скорость, отсутствие нативного движка Word, предсказуемость; в сообщениях пользователю явно указывается, что номер страницы для части проверок — **оценка по разрывам OOXML**, а не гарантированная печатная разметка.

## 2. Загрузка и парсинг .docx

### 2.1. Валидация входного файла

Перед парсингом проверяются имя и MIME/расширение (**`DocumentFileValidator`**), чтобы не подставлять произвольные файлы в POI. Неподдерживаемый формат — **`ValidationException`** с понятным текстом.

### 2.2. Защита ZIP и «ложные» zip bomb (Apache POI)

При открытии .docx POI использует **`ZipSecureFile`** с порогом сжатия **min inflate ratio**. Для **сильно сжатых** бинарных частей пакета (типично EMF в `word/media/`) стандартный порог может дать **ложное срабатывание** «zip bomb».

**Решение в коде:** перед `new XWPFDocument(inputStream)` вызывается  
`ZipSecureFile.setMinInflateRatio(0.001)` (константа `POI_MIN_INFLATE_RATIO_FOR_DOCX` в `DocxLoadService`).

**Смысл решения:** сохранить защиту от злонамеренных архивов, но разрешить легитимные тяжёлые вложения в учебных/отчётных документах.

### 2.3. Обход тела документа

Итератор **`doc.getBodyElementsIterator()`** обрабатывает:

- **`XWPFParagraph`** — основной текст, заголовки, подписи;
- **`XWPFTable`** — ячейки как отдельные абзацы с `ParagraphInfo` (в т.ч. для проверок списков и таблиц);
- **`XWPFSDT`** (структурированные документы — оглавление и т.п.) — рекурсивный разбор через `appendSdtContent`, чтобы не пропускать текст внутри SDT.

Для каждого абзаца строится **`ParagraphInfo`**: текст, стиль, выравнивание, шрифт/размер (из ранов и при необходимости из OOXML), отступы, межстрочный интервал, признаки списка, таблицы, **оценка страницы** (`pageIndex` / `pageEndIndex` через `PageLocator` и доработки `paragraphStartPageForContent` для разрывов в начале абзаца).

### 2.4. Секции и поля страницы

- **Секции** собираются из `w:sectPr` в конце абзацев и в теле документа; индексы абзацев с `sectPr` сохраняются в **`sectPrParagraphIndices`** — для проверок, которые должны знать «границы секции» (поля, нумерация страниц).
- **`w:pgMar`** переводится в сантиметры (twips → см, коэффициент `1/567` см на twip) в **`PageMargins`**.
- **`DocumentPageSettings`** включает список **`SectionPageInfo`** (размер страницы, поля, параметры `w:pgNumType`, флаги перезапуска нумерации).

### 2.5. Колонтитулы и нумерация страниц (подготовка к ФТ-12)

- Анализ **`XWPFHeaderFooterPolicy`**: default / first / even footer.
- Поиск поля **`PAGE`** в XML подвала (классические `instrText`, `fldSimple`, варианты из галерей Word «Page Numbers» в `w:sdt`).
- Собирается **`PageNumberingInfo`**: есть ли PAGE в подвале, выравнивание по центру (по XML `jc`), подозрение на точку после номера, признаки отдельных подвалов первой/чётной страницы.

**Здесь же была доработана логика под LibreOffice:** при включённом «одинаковое содержимое» в OOXML может существовать **отдельная часть подвала для чётных страниц без поля PAGE**, хотя номер визуально везде есть (поле живёт в default footer). Решение:

1. **«Пустой» подвал** (нет текста и нет PAGE) не считается «отдельным содержательным» подвалом для цели проверки (**`footerBodyMateriallyEmpty`**).
2. В **`Ft12PageNumberingChecker`**: если в **основном** подвале уже есть PAGE и поле PAGE в подвалах найдено, **не выдавать** сообщение про «отдельный подвал чётных без PAGE» — чтобы не наказывать артефакт разметки.

### 2.6. Таблицы, рисунки, подписи

- Таблицы: **`TableInfo`** с диапазоном индексов абзацев, подписью, страницей; связь с ближайшей строкой «Таблица N …».
- Рисунки: **`FigureInfo`** из inline-графики; привязка к подписям «Рисунок N …».

### 2.7. Старый формат .doc (опционально)

В `load()` поддерживается ветка **`hw`** через **`HWPFDocument`** для базового разбора; основной фокус этапа — **.docx**.

## 3. Ядро проверок (соответствие формулировке ТЗ)

Требования ТЗ сопоставлены с **функциональными требованиями ФТ** из методички:

| Тема ТЗ | Примеры правил в `checks-config.json` |
|--------|----------------------------------------|
| Обязательные разделы | **ФТ-4** — фиксированные заголовки (СОДЕРЖАНИЕ, ВВЕДЕНИЕ, …) |
| Нумерация разделов | **ФТ-5** |
| Начало с новой страницы | **ФТ-6** — оценка по `PageLocator` и страницам абзацев |
| Содержание | **ФТ-7** |
| Шрифты основного текста | **ФТ-8** (Times New Roman, кегль, цвет) |
| Абзацы (интервал, отступ, выравнивание) | **ФТ-9** |
| Поля страницы | **ФТ-10** — `w:pgMar` по секциям |
| Нумерация страниц | **ФТ-12** — поле PAGE, центр, сквозная нумерация vs `w:start` по секциям |

Дополнительно реализованы ФТ-11, 13–21 (заголовки, подписи к рисункам/таблицам, приложения, списки, формулы, библиография и т.д.) — расширение ядра за рамки минимальной формулировки, но в том же конвейере.

### 3.1. Конфигурация правил

Файл **`src/main/resources/checks-config.json`** задаёт:

- `id` правила (строчное имя, совпадает с `case` в `VkrChecksRunner`);
- `enabled` — можно отключить правило без перекомпиляции;
- `params` — числовые пороги (поля в см, отступы, кегль и т.д.);
- опционально `logDiagnostics` для тяжёлых проверок (ФТ-17, 18, 20) — вывод вспомогательной диагностики в колбэк.

**Почему JSON:** изменение норм или порядка проверок без правки Java-кода; удобно для согласования с преподавателем.

### 3.2. Параметры норм и `CheckSession`

Часть чекеров читает параметры через **`CheckRuleDefinition.fromRule`** / record-классы (`Ft10MarginsParams`, `Ft9ParagraphParams`, …). **`CheckSession.begin(config)`** привязывает текущее правило к потоку, чтобы **`Ft9MainParagraphChecker`** и аналоги брали актуальные `eps` из конфига.

### 3.3. Типичные сообщения и ограничения модели

- **Поля (ФТ-10):** сравниваются значения из **`w:pgMar`**, а не «как на линейке в Word». Если в XML нижнее поле 30.7 мм, а в редакторе кажется 20 мм, расхождение объясняется тем, что **источник истины для проверки — OOXML**, а не скриншот.
- **Нумерация страниц (ФТ-12):** проверяется наличие поля PAGE, выравнивание, отсутствие точки после номера, согласованность с **`w:start`** по секциям — **без рендера PDF**.

## 4. Сборка, Docker и окружение

### 4.1. Версия Java

В **`build.gradle`** заданы **`sourceCompatibility` / `targetCompatibility` = 21** (байткод Java 21), без отдельного **toolchain** Gradle.

**Причина:** при использовании `java { toolchain { languageVersion = 23 } }` в образе Docker был только **JDK 21**, Gradle пытался подтянуть JDK 23 через **Foojay** и получал **таймауты**; на хосте без JDK 21 toolchain тоже ломался. Решение: **компиляция с тем JDK, который запускает Gradle** (`--release 21`), что согласуется с **`eclipse-temurin:21-jdk`** и **`amazoncorretto:21`** в **Dockerfile**.

### 4.2. API

REST-слой принимает загрузку файла, вызывает **`CheckEngineService.validate`**, возвращает **`ValidationResult`** JSON (инфо о документе, сводка, список ошибок).

## 7. Фразы для отчёта «зачем так сделано» (готовые тезисы)

- Разделение **загрузки** и **правил** упрощает тестирование и изменение норм без изменения парсера.
- Использование **Apache POI** снижает стоимость внедрения по сравнению с коммерческими SDK; цена — **эвристики** вместо полной вёрстки.
- **JSON-конфигурация** правил отражает требование гибко подстраивать проверки под методичку.
- Правки **ФТ-10 / ФТ-12** основаны на том, что **валидация должна измерять файл**, а не ожидания пользователя с экрана редактора; при этом **ФТ-12** смягчён для типичных **LibreOffice**-артефактов, чтобы не было ложных срабатываний при корректном документе.
- **ZipSecureFile** настроен так, чтобы не блокировать **валидные** документы с тяжёлыми изображениями.
