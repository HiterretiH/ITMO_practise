## 1. Сообщение ФТ-10 про нижнее поле (ожидалось 20 мм, фактически ~30,7 мм)

**Что просили:** Объяснить, что значит формулировка про секцию, абзацы, «оценку страниц 1–12», несоответствие нижнего поля норме; как исправить и «на какой странице»; при этом в Word отступы визуально казались правильными на каждой странице.

**В чём была путаница:** Проверка смотрит не на экран и не на линейку, а на **значения в файле .docx** (нижнее поле в настройках страницы для секции — в терминах OOXML это атрибут нижнего поля в `w:pgMar` у секции). Норма по методичке — 20 мм; в XML оказалось ~30,7 мм (типично близко к дюймовым шаблонам вроде ~1,2″).

**Как объясняли решение:** Поля задаются **на секцию**, а не на отдельную физическую страницу; диапазон «стр. 1–12» в сообщении — **оценка** по разрывам в XML, а не указание «одной битой страницы». Путать **поля страницы** (макет) и **отступы абзаца** нельзя. Исправление — явно выставить нижнее поле 2 см в настройках страницы и пересохранить тот же файл, который гоняется через проверку.

**Итог для отчёта:** Расхождение «вижу нормально» vs «сканер ругается» объясняется тем, что источник истины для автоматической проверки — **структура OOXML**, а не визуальное восприятие.

## 2. Сборка Docker: падение Gradle на этапе `bootJar`

**Что не работало:** Команда вроде `docker compose build --no-cache` падала на шаге запуска Gradle внутри образа. Сообщение: не удаётся разрешить toolchain для компиляции — нужна **Java 23**, в Linux-образе её нет, автозагрузка через Foojay завершалась **`Read timed out`**.

**В чём суть проблемы:** В Gradle было задано требование **toolchain с languageVersion 23**, а в Dockerfile использовался образ с **JDK 21**. Gradle пытался **скачать** подходящий JDK с внешнего репозитория (Foojay) — таймаут сети из контейнера. Отдельно на хосте при попытке перейти на toolchain 21 тоже всплывал таймаут Foojay, если локально не было нужной версии.

**Как решали:** Отказ от жёсткого toolchain в пользу явного целевого уровня языка **21** (`sourceCompatibility` / `targetCompatibility`), чтобы компилятор брался из **того JDK, которым уже запущен Gradle** (в Docker это JDK из образа, без докачки). Это снимает конфликт «образ 21 / в build.gradle просили 23» и уменьшает зависимость от внешней загрузки JDK при сборке.

**Итог для отчёта:** Проблема была **инфраструктурной** (несовпадение версий + нестабильная сеть для авто-провижининга), а не логикой приложения.

## 3. П. 4.3.1: нумерация страниц в LibreOffice и ложное «чётные без PAGE»

**Что просили:** Подробный гайд **куда нажимать в LibreOffice**: сквозная нумерация, внизу по центру, без точки после номера; сообщение проверки про отдельный подвал чётных страниц без поля PAGE (стр. 2, 4, 6…).

**Что объясняли по редактору:** Вариант с **одним** нижним колонтитулом на все страницы; вариант с разными подвалами — тогда номер нужно вставить **и в чётный** подвал; поле через **Вставка → Поле → Номер страницы**, выравнивание по центру; убрать точку после номера, если она не часть поля.

**Что не сходилось с ожиданиями пользователя:** В интерфейсе стояла галочка **«одинаковое содержимое на левой и правой / на первой странице»**, визуально номера везде были, а проверка всё равно писала про **отдельный подвал чётных без поля PAGE**.

**В чём была причина (смысловой уровень):** В .docx колонтитулы — отдельные части пакета. **LibreOffice** часто оставляет **отдельный XML-подвал для чётных страниц**, даже когда в UI «всё одинаково»; в этой части **нет поля PAGE**, а номер фактически задаётся в **`default` подвале**. Старый алгоритм проверки смотрел на наличие части «чётный подвал» и требовал в ней поле PAGE — отсюда **ложное срабатывание** при корректном документе.

**Как решали в коде (идея, без привязки к именам классов):**  
- Считать «отдельный подвал» значимым только если в нём **есть реальное содержимое** (не пустой файл).  
- Если в **основном** подвале уже есть поле PAGE и нумерация в подвалах в целом обнаружена, **не ругаться** на отсутствие PAGE именно в «чётной» части — так как это типичный артефакт LibreOffice при общем номере снизу.

**Итог для отчёта:** Конфликт «редактор говорит одно, XML — другое»; пришлось **смягчить правило** под реальные экспорты LibreOffice, не отменяя строгую проверку для случаев, когда чётный подвал реально отличается и без номера.

## 5. Ранние этапы разработки (что обсуждалось «до» и параллельно)

Ниже — **не дословный лог**, а пересказ из более ранних сессий: что **не работало**, что **просили**, как **разруливали**. К коду в репозитории это относится как к истории решений.

### 5.1. Запуск Gradle / Java на Windows

**Проблема:** В `build.gradle` изначально требовался **JDK 21** (toolchain), а в системе был **JDK 23** — Gradle не находил «свою» 21-ю версию без отдельной установки.

**Что делали:** Временно поднимали версию toolchain под установленный JDK **или** объясняли, как выставить `JAVA_HOME` и запускать из `backend` через `gradlew.bat`. Позже эта линия пересеклась с Docker (см. §2): toolchain + удалённая докачка JDK давали отдельные боли.

### 5.2. Парсинг .docx и «что уже есть»

**Запрос:** Разобрать `task.mdc`, сказать **что уже парсится**, что нет; запустить **`DocxLoadDebugMain`** на `test-input/test.docx` без выдумывания новых файлов.

**Вывод по сути:** Парсер уже тянул многое из OOXML (абзацы, стили, поля, секции, колонтитулы, таблицы, рисунки и т.д.), но **страницы** изначально были заглушкой / `null`, **ядро проверок в API** — не полный конвейер. Это потом эволюционировало в текущий `CheckEngineService` + конфиг правил.

### 5.3. ФТ-6 и оценка страниц

**Запрос:** «Вычленить страницы», реализовать **ФТ-6** (начало разделов с новой страницы) — казалось «изи».

**Что сделали по идее:** Ввели эвристику по **разрывам в OOXML** (`w:br` page, `lastRenderedPageBreak`, разрыв секции «следующая страница»), а не полный движок вёрстки Word.

**Когда «не ловит» ошибку (важно для отчёта):** Если в файле **нет** ни явных разрывов, ни сохранённых Word’ом `lastRenderedPageBreak`, «страницы» схлопываются в одну-две **условные** — проверка **не увидит**, что заголовок визуально не с новой страницы. Рекомендовали пересохранять документ в Word, чтобы в XML попала вёрстка.

**Дополнительно:** Учли разрывы **в начале абзаца до текста** (иначе ФТ-6 ошибочно думал, что заголовок на той же странице).

### 5.4. ФТ-7 и оглавление — длинная цепочка сбоев

1. **Нет заголовка «СОДЕРЖАНИЕ» в потоке абзацев** — в тестовом документе сначала шёл другой раздел; полная проверка оглавления **не запускалась**. Объясняли разницу между «0 замечаний» и «проверка пропущена».

2. **«СОДЕРЖАНИЕ» vs «ОГЛАВЛЕНИЕ»** — расширили поиск заголовка блока; нормализация пробелов/переносов.

3. **TOC1 / TOC2 / TOC3** — объясняли: это **уровни стилей встроенного оглавления Word**, не «магические буквы»; в чужом шаблоне стили могут называться иначе — тогда по одним только именам стилей оглавление не находилось.

4. **Главный технический затык:** оглавление лежало внутри **SDT** (`w:sdt`, галерея Table of Contents) с полем TOC, а обход тела шёл только по верхнеуровневым параграфам/таблицам — **весь блок оглавления не попадал в модель**. Пока не добавили обход **SDT**, строки вроде 3.1–3.3 в логе просто **отсутствовали**, и сравнить «лишнее в содержании» с текстом было невозможно. Доступ к внутренним элементам в используемой версии POI частично шёл через **reflection**.

5. **Эвристика «самая длинная цепочка»** ломалась на битых строках оглавления (например, оторванный номер страницы) — уточняли: цепочка **после заголовка**, пропуск «шума», не обрывать блок на пустых/странных строках.

### 5.5. ФТ-4 / ФТ-5 (разделы и нумерация)

- **ПРИЛОЖЕНИЕ:** граница слова **`\b` в Java для кириллицы** вела себя не как в «английском» regex — паттерн для приложений поправили.

- **ФТ-5:** сначала обсуждали гибкие правила (пропуски, опечатки типа 3.52), потом по запросу вернули **строгий** порядок подпунктов **N.M → N.(M+1)**.

- **Многоуровневая нумерация (2.1.1):** раньше пары с числом сегментов ≠ 2 игнорировались; добавили проверку **иерархии** «как в оглавлении» (первый подпункт после главы — `.1`, углубление — `…1` после родителя, на том же уровне — +1 к последнему сегменту, подъём уровня — следующий номер у предка).

- **Несоответствие теста и ожиданий:** ошибки 3.52 / 2.52 в **оглавлении** без таких же заголовков в **теле** с нужным outline — ФТ-5 смотрит на заголовки основного текста; это отдельно объясняли.

### 5.6. ФТ-8 / ФТ-9 (основной текст)

**Запрос:** Проверять только **основной текст**, не таблицы/формулы и т.д.

**Проблемы по пути:**

- Строки оглавления без `outlineLevel`, но с «похожим» видом — ФТ-9 ругался на весь TOC; добавили **исключения** (стили TOC, эвристика строки с табом/пробелами перед номером страницы).

- **`\b` снова** — подписи «Рисунок …» на кириллице не отфильтровывались; перешли на regex **без** `\b` для таких шаблонов.

- **«Ничего не находит» при смешанном шрифте:** агрегация шрифта шла по **числу прогонов**, а не по символам — побеждал «частый» прогон, хотя кусок текста был другим шрифтом. Добавили **по-прогонную** проверку с чтением `w:rPr` из XML и учётом наследования стиля.

**Запрос:** На каждое нарушение — **конкретное сообщение** (что ожидалось, что фактически, превью текста) — доработали формулировки ФТ-8/ФТ-9.

### 5.7. Кодировка логов и файлы `right.txt` / `wrong.txt`

**Проблема:** При перенаправлении вывода в файл на Windows **кракозябры** (не UTF-8).

**Решение:** Писать лог через настроенный **файловый аппендер в UTF-8**, JVM-аргументы для кодировки в Gradle; по умолчанию **`test2` в имени** → `wrong.txt`, иначе `right.txt`; убрать размножение `test_run*.txt`.

### 5.8. Git: rebase, конфликты, stash

**Проблема:** `git pull --rebase`, конфликты, **заблокированный** `test2.docx` (открыт в Word), неполная очистка после stash.

**Суть конфликтов:** **modify/delete** для `output/right.txt` и `wrong.txt` — на `origin` их **убрали из репозитория** (gitignore), локально они ещё трогались. Решение: оставить **удаление** как на remote.

**Пуш** с машины ассистента не выполнялся (нет учётки) — описали, что сделать локально.

### 5.9. Zip / «zip bomb» при открытии .docx

**Проблема:** Apache POI мог отказываться открывать документ с **сильно сжатыми** бинарными вложениями (например EMF в `word/media/`) из‑за защиты от zip bomb.

**Решение:** Снизить порог **min inflate ratio** у `ZipSecureFile` перед открытием — не отключая защиту полностью.

### 5.10. Деплой, таймауты, «в логах пусто»

**Симптомы:** Долгий `/v1/validate`, таймаут на клиенте; в логах Spring после `DispatcherServlet` **тишина**; или validate «не получает ответа».

**Что разбирали:**

- **Кэш `checks-config.json`** — раньше мог парситься на каждый запрос; кэш при старте уменьшает лишнюю работу.

- **Логи с разбивкой** `load` vs `checks` (мс), логи воркера очереди, `copyMs` при приёме файла.

- **Ubuntu / Docker / сеть:** не «магия ОС», а CPU, диск, очередь, прокси, буферизация тела POST, таймауты nginx — объясняли, **что смотреть** в логах (`copyMs`, `docx POI parse done`, `validation pipeline`, при DEBUG — время каждого ФТ).

- **Пустые логи при запросе:** частая причина — фронт бил **не туда**: в конфиге был **`http://localhost:8080`** при открытии UI **с другого хоста** — `localhost` оказывался **комьютером пользователя**, не сервером. Исправление: **относительный** `/v1` + прокси dev-server / docker.

- **405 на GET в браузере** к `/v1/validate` — **нормально**: endpoint только **POST**; 405 подтверждает, что сервер живой, но метод не тот.

- Фильтр **диагностики HTTP** (`>>>` / `<<<` в логах) — чтобы видеть: запрос **вошёл** в JVM и **вышел** ли ответ, или завис при приёме тела.

### 6.1. ФТ-10 и «каждая страница»

**Запрос:** проверять поля **для каждой страницы** и точнее указывать, где ошибка.

**Суть ответа (идея):** в .docx поля задаются **на секцию** (`w:pgMar`), а не на каждую физическую страницу; без движка вёрстки **нельзя** привязать то же значение к «стр. 5» vs «стр. 6», если в XML одна секция. Сообщения строятся как **оценка** по разрывам и диапазону абзацев — это ограничение подхода, не «лень».

### 6.2. ФТ-11 (заголовки): полужирный vs semibold, CAPS, аббревиатуры, переносы

**Что не устраивало:**

- Ожидание, что **полужирный** в смысле ТЗ = **semibold**, а не `bold` в XML; LibreOffice **Ctrl+B** может давать другое отображение, чем ожидал чекер.
- Сообщения в духе «все заголовки плохие» вместо **сравнения вариантов** (главы vs подразделы).
- **`caps = false` в логе**, хотя визуально текст прописными — из-за того, как Word хранит капс (стиль/XML vs фактический текст).
- Пункты **task.mdc (36–38)**: переносы, аббревиатуры, абзацный отступ — нужно применять **только к заголовкам разделов**, не ко всем подряд.
- **test.docx** «тоже не соответствует» при ожидании «идеального» right.txt — разбор структуры документа и стилей vs ожидания пользователя.
- **Перенос слова в заголовке** (одно слово на двух строках / дефис переноса): вопрос, можно ли детектить без рендера — обсуждение **ограничений** (в OOXML нет готовой «картинки строк»).

### 6.3. ФТ-12: сквозная нумерация, подвалы, «цифры в углу»

**Запросы по цепочке:**

- Нумерация **последовательная на каждой странице**, старт не обязательно с 1, но **целое > 0**.
- Объяснение: **почему вообще бывает «нет номера на чётных»** в сообщениях; что такое **перезапуск** нумерации по секциям; что такое **отдельный подвал первой страницы** в OOXML.
- Акцент на формулировках: **всегда снизу, по центру, без точки** после номера.
- **Проблема:** в «плохом» файле нумерация скачет (пример: ряд чисел **7 9 1 52 11…** визуально в подвале), а в отчёте **нет** явной строки «вот что нарисовано в подвале».

**Ключевая мысль для отчёта:** поле **`PAGE`** в колонтитуле задаёт **формулу**, а не константу. Программа **не читает «какое число напечатано»** после вёрстки Word — только XML. Если пользователь **вручную подменил числа** в подвале без нормального поля или с поломанной логикой секций, увидеть «7, 9, 52» как факт печати **из .docx тем же способом нельзя**; альтернатива — экспорт в PDF и другой пайплайн (отдельно обсуждалось как **предел возможностей** текущего подхода). Отсюда и фрустрация «почему не пишется нигода».

## 7. Сводка именно **этой** переписки: проблема → в чём дело → что поменяли

Ниже — не дублирование §5–6 из старых сессий, а **склейка того, что обсуждалось в текущем чате** (включая запросы «а ещё?» и дополнения к файлу): где **код был неправ**, где ошибки **не в коде**, а в ожиданиях или в данных документа.

| Симптом / вопрос | Что было «не так» | Как решали |
|------------------|-------------------|------------|
| ФТ-10: нижнее поле 20 мм в норме, в отчёте ~30,7 мм; «на экране всё ок» | **Не баг парсера.** Проверка читает **`w:pgMar` в OOXML**; визуальная линейка и отступы абзаца — другие сущности. В файле реально было другое нижнее поле (или секция не та). | Объяснение пользователю: править **макет страницы** и пересохранять `.docx`. **Код ФТ-10 не меняли** под этот запрос. |
| `docker compose build` падает: toolchain Java 23, в образе JDK 21, Foojay **Read timed out** | **Конфиг сборки:** `build.gradle` требовал **toolchain 23**, Docker — **JDK 21**; Gradle пытался **скачать** JDK, сеть из контейнера таймаутилась. На хосте отдельно toolchain 21 тоже ломался без локального JDK / при таймауте Foojay. | Убрали **toolchain**, задали **`sourceCompatibility` / `targetCompatibility` = 21** — компиляция идёт тем JDK, что уже запускает Gradle (в образе — 21). |
| ФТ-12: галочка LibreOffice «одинаковое содержимое», но ошибка «чётные без PAGE» | **Логика проверки:** POI видит **отдельную часть подвала для чётных страниц** в пакете `.docx`; в ней часто **нет поля PAGE** (номер живёт в default), либо часть **пустая**, но всё равно «есть». Старый алгоритм трактовал это как нарушение. | В загрузчике: подвал считать «отдельным» только если в нём **есть содержимое** (не пустой). В ФТ-12: если **в основном подвале уже есть PAGE** и поле PAGE в подвалах найдено — **не выдавать** замечание про чётный подвал без PAGE. |
| Нужен текст для отчёта по практике для DeepSeek | Сначала сделали файл с **описанием классов и архитектуры** — пользователю нужен был **не он**, а **пересказ переписки** (проблемы, как чинили). | Добавлен **`CHAT_RECAP_FOR_REPORT.md`**, затем расширен §5–6 из **транскриптов** старых сессий. |
| «А ещё?» — что было в старых чатах | История не в одном месте. | Вынесено в §5–6: Gradle/Windows, ФТ-6…13, SDT/оглавление, regex/кириллица, UTF-8 логов, git, zip bomb, деплой/логи/localhost. |

### Что в **коде** реально исправляли в рамках этой ветки обсуждения

1. **`build.gradle`** — отказ от `java { toolchain { languageVersion = 23 } }` в пользу `sourceCompatibility`/`targetCompatibility` **21** (и актуализация комментария в **Dockerfile**).
2. **`DocxLoadService`** (анализ колонтитулов) — учёт **пустого** подвала чётных/первой страницы как несущественного для флагов «отдельный подвал».
3. **`Ft12PageNumberingChecker`** — пропуск ложного срабатывания «чётный подвал без PAGE», когда **default-подвал уже с доверенным PAGE** и нумерация в подвалах обнаружена.
4. **Тест** — кейс, что при такой комбинации флагов сообщение про чётные не должно появляться.

### Что **не** считали ошибкой кода

- Расхождение **полей по ФТ-10** с «как кажется в Word» — до уточнения правил норма сравнивается с **XML**, не со скриншотом.
- Сообщения ФТ-12 про подвалы **до** правки — следствие **строгой модели OOXML** (несколько `footer*.xml`), а не «сломанный» LibreOffice.
